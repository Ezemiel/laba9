## 1-20 шага

1. По прошлой работе переключимся также через команду `git switch` переключимся, а лучше создадим новую ветку для выполнения работы для оценки 4.

2. Изменим файлик [BubbleSort.c](lab9_git/BubbleSort.c) так, чтобы он включал в себя следующие изменения:
 Код содержимого файлика:
 ```
#include <stdio.h>
 void bubbleSort(int arr[], int n) {
    for (int i = 0; i < n-1; i++) {
        for (int j = 0; j < n-i-1; j++) {
            if (arr[j] > arr[j+1]) {
                int temp = arr[j];
                arr[j] = arr[j+1];
                arr[j+1] = temp;
            }
        }
    }
}

int main() {
    int a[] = {4, 2, 0, 8, 5};
    int n = sizeof(a)/sizeof(a[0]);
    
    bubbleSort(a, n);
    
    printf("Sorted array: ");
    for (int i = 0; i < n; i++) {
        printf("%d ", a[i]);
    }
    printf("\n");
    
    return 0;
}
 ```

3. Анализируем изменения с помощью нашей любимой команды `git diff`. Вывод такой
```
Те что красные помеченные строки те удаленные;
Те что зелёные мы добавили например с файлом BubbleSort.c
```

4. Также проверим stagged (это что мы добавили в staging area — это промежуточная зона в Git, куда вы добавляете изменения с помощью git add, чтобы подготовить их к следующему коммиту) файлы с помощью команды `git diff` и подпараметра `--staged` - теперь мы видим только добавленные файлы. Вывод:
```
Пусто так как никакие файлы не были добавлены через git add
```

5. Добавим также нашу сортировку через команду `git add BubbleSort.c`. Теперь в staging area содержится только файл BubbleSort.c с изменениями.

6. Ещё раз вводим git diff, видим всё тоже самое, что и вводили до этого.

7. А если введем  `git diff --staged`, то уже увидим наш файл что находится в staging area и только его, видим что удалилось из него и что добавилось.

8. Удалим что-нибудь из файла BubbleSort.c например строку из массива int a[] = {4, 2, 0, 8, 5}; и удалим из него цифру 0;

9. Смотрим ещё раз изменения через `git diff` и `git diff --staged`, через 1-ю команду видим что она увидела что 0 удалился, так как 2 команда сохранила файл до этого и показывает так, как он будет отправлен на удалённый репозиторий;

10. Проверяем также статус ветки, видим что файл инициализирован и подсвечивается зелёным цветом.

11. Попробуем командой `git restore` отменить индексацию нашего файлика.
* git restore - восстанавливает файлы в рабочей директории, но не в staging area. 

12. Как мы видим через команду `git status`, файл не пойдёт в репозиторий, но не трогает его последние изменения в директории.

13. Ещё раз индексируем его всеми известной командой и потом отправляем коммит. Вывод сообщения:
```
[ochenka_4 commit-hash] Усовершенственный пузырьковый метод
1 file changed, 23 insertion(+), 23 deletion(-)
```

14. После чего просмотрим через `git log --oneline --graph` и увидим наш коммит

15. Добавим также в файлик сортировки [BubbleSort.c](lab9_git/BubbleSort.c) через команду echo какой-нибудь код например `printf("hello git\n");`.

16. Просмотрим статус ветки, видим что файл не инициализирован и подсвечивается красным цветом.

17. Через команду cat также содержимое просмотрим файлик [BubbleSort.c](lab9_git/BubbleSort.c), видим то что мы писали в терминал есть в файле.
Вывод:

18. Пробуем через команду `git restore` отменить что писали последнее, cейчас она удалит наш код что мы добавили и из-за этого что он остался без изменений он будет у нас инициализирован как и прежде.

19. Проверяем статус ветки и да его нету в списке тех файлов что не будут идти в репозиторий после коммита.
